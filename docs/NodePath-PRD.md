# NodePath 产品需求文档（PRD）

> 版本：V1.0  
> 日期：2026-07-20  
> 产品形态：移动端优先的题目驱动式编程学习网站  
> 当前技术栈：Next.js 16、React 19、TypeScript、Tailwind CSS、Three.js、Supabase（接入中）

## 1. 文档目的

本文档用于明确 NodePath 后续阶段的产品定位、用户需求、功能范围、交互方式、数据结构和验收标准，为产品设计、课程制作、前端开发、后端开发和测试提供统一依据。

NodePath 当前已经完成 Node.js 与 Next.js 两条学习路线、182 个可交互案例、阶段星图、3D 运行舱、粒子云、学习进度及游戏化反馈等基础能力。下一阶段不以继续大量扩充课程数量为首要目标，而是增强题型丰富度、移动端学习体验、阶段项目连续性和学习数据沉淀能力。

## 2. 产品背景

传统编程学习网站通常依赖长篇阅读、视频课程或桌面端代码编辑器。用户使用手机时，存在以下问题：

- 手机屏幕不适合长时间输入和编辑代码。
- 用户在通勤、休息等碎片时间里难以搭建开发环境。
- 单纯阅读知识点缺少互动，注意力难以维持。
- 普通选择题容易沦为记忆答案，无法训练代码理解能力。
- 抽象的运行机制难以通过静态文字形成直观认知。
- 学完单个知识点后，用户不知道它如何应用到真实项目。

NodePath 希望通过“短知识点＋代码判断＋定向反馈＋运行可视化＋连续式项目”的方式，让用户在手机上也能进行有效的编程思维训练。

## 3. 产品定位

### 3.1 一句话定位

NodePath 是一个移动端优先的题目驱动式编程学习平台，通过多类型代码选择题、阶段项目和 3D 运行可视化，帮助用户理解代码、判断运行逻辑并建立技术栈心智模型。

### 3.2 核心价值

- 随时学习：无需安装开发环境，使用手机即可完成课程。
- 主动思考：先判断、再反馈，避免只被动阅读答案。
- 理解代码：重点训练代码阅读、结果推演、错误诊断和方案选择。
- 项目迁移：通过连续任务把零散知识点组合成项目能力。
- 直观记忆：使用 3D 场景、粒子流和节点关系展示抽象运行机制。
- 持续成长：记录进度、错题、掌握度和阶段成就，支持跨设备学习。

### 3.3 当前阶段不做的事情

- 不把 NodePath 建设成手机端在线 IDE。
- 不要求用户在手机上输入大段代码。
- 不执行用户提交的任意 Node.js 代码。
- 不在 Next.js 应用进程中运行不可信代码。
- 不以增加纯装饰性动画作为主要迭代方向。
- 不在本阶段建设社区、直播或复杂社交系统。

## 4. 目标用户

### 4.1 核心用户

- JavaScript 基础较弱，希望入门 Node.js 或 Next.js 的学习者。
- 已掌握基础语法，希望系统理解工程运行机制的开发者。
- 主要使用手机，在通勤、午休和睡前进行碎片化学习的用户。
- 不方便随时打开电脑和开发环境，但希望维持学习节奏的用户。

### 4.2 用户特征

- 单次学习时间通常为 5～20 分钟。
- 更适合点击、选择、滑动和排序，不适合大量键盘输入。
- 需要快速获得反馈，并理解错误原因。
- 希望看到明确的阶段目标、完成进度和成长反馈。
- 对视觉化、游戏化体验有兴趣，但仍以知识学习为主要目的。

## 5. 产品目标

### 5.1 近期目标

- 建立可扩展的多题型课程数据结构。
- 在现有“代码预测结果题”之外，上线“根据结果选择代码”题型。
- 完善手机端代码选项的阅读、比较和选择体验。
- 支持一个知识点包含多道、不同类型的问题。
- 保留 authored trace 确定性运行模型，并按题目结果播放可视化过程。

### 5.2 中期目标

- 上线错误诊断、代码修复、代码补全、执行顺序等题型。
- 将阶段项目升级为多步骤、连续式选择项目。
- 接入 Supabase 登录、跨设备进度、错题本和学习记录。
- 建立知识掌握度与智能复习机制。
- 让 3D 和粒子效果与代码执行语义形成稳定映射。

### 5.3 长期目标

- 扩展 React、TypeScript 等更多学习路线。
- 建立课程内容管理与发布工具。
- 根据用户薄弱知识点动态生成复习路径。
- 提供综合挑战、限时训练和学习报告。
- 在桌面端按需提供受控的轻量代码实验能力，但不影响移动端核心体验。

## 6. 核心学习闭环

每个知识点采用以下学习流程：

```text
知识点讲解
→ 阅读代码或目标结果
→ 完成一种或多种选择题
→ 查看选项对应的定向反馈
→ 正确后播放运行过程
→ 查看终端输出和知识总结
→ 更新掌握度与学习进度
```

阶段项目采用以下流程：

```text
项目背景
→ 明确当前任务
→ 连续完成多个项目决策
→ 每一步获得代码与运行反馈
→ 逐步组装完整项目
→ 展示项目最终结果
→ 解锁阶段徽章和项目总结
```

## 7. 题型系统

### 7.1 题型总览

| 题型标识 | 题型名称 | 题目提供 | 用户选择 | 训练目标 | 优先级 |
| --- | --- | --- | --- | --- | --- |
| `prediction` | 结果预测 | 代码 | 正确输出或行为 | 阅读与推演代码 | P0，已具备 |
| `implementation` | 代码实现 | 目标结果或需求 | 正确代码方案 | 从目标反推实现 | P0 |
| `diagnosis` | 错误诊断 | 错误代码、日志或现象 | 错误原因 | 定位问题 | P1 |
| `repair` | 代码修复 | 错误代码 | 正确修改方案 | 修复 Bug | P1 |
| `completion` | 代码补全 | 带缺口的代码 | 正确代码片段 | 理解上下文 | P1 |
| `execution-order` | 执行顺序 | 同步或异步代码 | 正确执行顺序 | 理解运行机制 | P1 |
| `best-practice` | 最佳实践 | 需求与多种可运行方案 | 更合理的方案 | 建立工程判断 | P2 |
| `concept-match` | 概念匹配 | 使用场景 | 合适的 API 或概念 | 技术选择 | P2 |
| `equivalent-code` | 等价代码 | 已知代码 | 行为相同的实现 | 理解语义 | P2 |
| `sequence` | 步骤排序 | 若干代码步骤或运行阶段 | 正确顺序 | 建立过程模型 | P2 |

### 7.2 结果预测题

系统展示代码，用户选择正确的运行结果、执行顺序或行为描述。

需求：

- 支持文本、日志、数组、对象、异常和执行顺序等答案形式。
- 用户提交前不显示正确答案。
- 每个错误选项必须对应一个常见误区。
- 答错后显示当前选项对应的解释，不直接替用户完成推理。
- 答对后播放 authored trace、终端日志和知识总结。

### 7.3 代码实现题

系统先展示目标结果、功能要求或业务需求，再提供多段代码供用户选择。

示例：

```text
目标：将 [1, 2, 3] 转换为 [2, 4, 6]
```

用户需要在 `map()`、`filter()`、`reduce()` 等代码方案中选择能够得到目标结果的实现。

需求：

- 每个代码选项支持独立语法高亮。
- 对相似方案优先突出差异行，减少手机端重复阅读。
- 支持点击展开完整代码。
- 支持折叠相同上下文，只显示关键差异。
- 每个选项必须提供选择后的定向反馈。
- 正确后展示所选代码如何一步步产生目标结果。

### 7.4 错误诊断题

系统展示代码、错误日志或异常现象，用户判断根本原因。

需求：

- 题目应包含足够的诊断线索。
- 干扰项应对应真实常见误判。
- 反馈需要区分“现象”“直接原因”和“根本原因”。
- 可视化应突出错误发生的节点或数据流中断位置。

### 7.5 代码修复题

系统展示存在问题的代码，用户从若干修改方案中选择正确修复。

需求：

- 优先展示修改前后的差异，而不是重复展示全部代码。
- 支持删除、替换、移动或新增代码片段等修复形式。
- 答对后展示修复前后运行行为对比。
- 答错后解释该修改为何仍不能解决问题或会引入什么新问题。

### 7.6 代码补全题

系统展示缺少一处或多处代码的上下文，用户选择正确片段。

需求：

- 缺口应明确标记。
- 选项插入后提供即时预览，但提交前不判断正误。
- 手机端优先采用点击插入，不要求拖拽。
- 一道题原则上不超过三个缺口。

### 7.7 执行顺序题

系统展示事件循环、Promise、定时器、请求链路等代码，用户选择正确顺序。

需求：

- P1 阶段先使用单选顺序方案。
- P2 阶段可支持点击排序。
- 答案提交后，用节点和粒子流逐步播放真实顺序。
- 播放过程必须能暂停、重播和逐步查看。

## 8. 题目内容规范

### 8.1 每道题必须包含

- 稳定的题目 ID。
- 所属课程、阶段和知识点。
- 明确的题型。
- 问题说明。
- 必要的代码、日志、目标输出或场景材料。
- 2～4 个选项。
- 唯一正确答案；多选题在后续版本单独定义。
- 每个选项独立的定向反馈。
- 正确答案解析。
- 至少三条知识总结。
- 难度和预计作答时间。
- authored trace 或其他确定性运行反馈。
- 官方或可信来源及校验日期。

### 8.2 干扰项设计原则

- 错误选项必须来自真实误区，不能使用明显荒谬答案凑数。
- 不应通过选项长度、语气或格式暗示正确答案。
- 同一知识点的不同题目不重复使用完全相同的干扰逻辑。
- 代码选项应尽量保持结构相近，让用户关注关键差异。
- 初级题侧重概念区别，高级题侧重边界条件与工程权衡。

### 8.3 反馈设计原则

答错反馈需要包含：

- 用户选择的方案实际会发生什么。
- 该判断混淆了什么概念。
- 建议重新关注的代码或条件。

答对反馈需要包含：

- 为什么正确。
- 关键代码如何产生结果。
- 可迁移到其他场景的规则。

反馈不得只显示“回答正确”或“回答错误”。

## 9. 单课结构

### 9.1 建议结构

每个知识点由以下内容组成：

1. 知识点标题和预计用时。
2. 简短概念讲解。
3. 一个核心代码案例。
4. 1～3 道不同题型的问题。
5. 运行可视化和终端反馈。
6. 三条以上知识总结。
7. 下一知识点入口。

### 9.2 题量规则

- 入门知识点：1～2 道题。
- 核心知识点：2～3 道题。
- 容易混淆的知识点：至少包含一道诊断题或修复题。
- 项目前置知识点：至少包含一道代码实现题。
- 单课移动端目标完成时间控制在 5～10 分钟。

### 9.3 完成规则

- 用户完成规定的必答题后，知识点记为完成。
- 首次答错不阻断继续学习，但记录为待复习。
- 必答题完成后才能解锁知识总结和完整运行回放。
- 是否要求全部答对可由课程配置决定，默认允许纠错后完成。

## 10. 阶段项目

### 10.1 产品形态

阶段项目采用“连续式选择项目”，不要求用户手动编写完整代码。用户通过一系列代码选择、错误诊断和工程决策，逐步完成一个可理解的项目。

### 10.2 项目步骤示例

以“任务 REST API”为例：

1. 根据需求选择数据结构。
2. 选择正确的路由设计。
3. 选择请求参数校验代码。
4. 根据错误日志诊断 500 问题。
5. 选择正确的异步处理方式。
6. 选择超时和错误处理方案。
7. 查看组合后的完整项目和运行结果。

### 10.3 功能需求

- 每个阶段项目包含 5～10 个连续任务。
- 每一步明确展示项目当前状态、当前目标和已完成步骤。
- 前一步选择会影响当前展示的代码，但核心学习路径保持可恢复。
- 错误选择提供反馈并允许重新选择，不导致整个项目失败。
- 支持中途退出并恢复到上次步骤。
- 最后展示完整代码结构、运行过程和知识点复用关系。
- 完成后解锁阶段徽章，并写入项目完成记录。

## 11. 移动端交互需求

### 11.1 基本原则

- 主要操作必须能通过单手点击完成。
- 核心内容不能依赖悬停操作。
- 不要求横屏才能完成课程。
- 代码阅读区支持横向滚动，但正文页面不得整体横向溢出。
- 重要按钮需要位于拇指易触达区域。
- 视觉效果不能遮挡题目、代码、反馈和按钮。

### 11.2 代码选项组件

- 默认采用纵向卡片，避免四段代码并排压缩。
- 卡片摘要优先显示差异行。
- 点击卡片可展开完整代码或进入全屏比较。
- 支持在代码选项之间左右滑动，但必须同时保留清晰的点击选择方式。
- 选中状态需要同时使用颜色、图标和边框表达，不能只依赖颜色。
- 提交答案前允许修改选择。
- 提交答案后锁定本次选择，用户点击“重新作答”后再解锁。

### 11.3 桌面端增强

- 桌面端可以使用双列代码对比。
- 可显示更大的 3D 运行舱和知识关系图。
- 桌面端增强不能改变题目答案或学习规则。

## 12. 3D 与粒子可视化需求

### 12.1 定位

3D、粒子云和空间场景是教学反馈层，不只是装饰背景。它们用于展示代码执行顺序、数据流向、模块关系、并发状态和知识连接。

### 12.2 语义映射示例

| 学习主题 | 可视化表达 |
| --- | --- |
| 事件循环 | 粒子进入 microtask、timer、I/O 等队列并按顺序释放 |
| HTTP | 请求粒子进入服务器节点，经过路由和处理后返回响应 |
| Stream | 粒子密度表示数据流量，阻塞或堆积表示背压 |
| Worker | 多个空间节点表示主线程和 Worker，粒子表示消息传递 |
| 模块系统 | 节点连接表示 import、export 和依赖关系 |
| 实时通信 | 连接线表示客户端，粒子表示心跳、广播和消息 |
| 测试 | 测试光束依次命中功能节点，通过或失败形成不同反馈 |
| 诊断 | CPU、Heap、GC 等指标围绕执行核心变化 |

### 12.3 运行状态

- `idle`：场景保持低强度待机状态。
- `wrong`：突出冲突节点或错误路径，不使用过强惩罚动画。
- `running`：按照 authored trace 推进节点和粒子状态。
- `success`：点亮知识节点并触发完成反馈。
- `review`：允许用户逐帧查看和重播。

### 12.4 性能与降级

- 移动端默认优先保证题目流畅度。
- 容器过窄、WebGL 不可用或用户启用减少动态效果时，使用 2D/fallback。
- fallback 必须保留运行顺序、节点关系和日志语义。
- 动画不得造成页面位移、顶部留白或操作阻塞。
- 页面切换和重新作答时必须取消旧动画。

## 13. 用户与学习进度

### 13.1 未登录用户

- 可以浏览课程并完成题目。
- 进度保存在浏览器本地。
- 登录入口需要说明跨设备同步价值。
- 登录后应支持把本地进度合并到云端。

### 13.2 登录用户

- 跨设备同步课程进度。
- 保存每道题的作答结果和首次作答情况。
- 保存阶段项目当前步骤。
- 查看错题本、薄弱知识点和最近学习记录。
- 查看课程完成率、连续学习天数和已解锁成就。

### 13.3 进度数据

至少记录：

- 用户 ID。
- 课程 ID、阶段 ID、知识点 ID、题目 ID。
- 是否完成。
- 首次答案是否正确。
- 尝试次数。
- 最近选择答案。
- 首次作答时间和最近作答时间。
- 当前项目步骤。
- 题目是否加入待复习。
- 知识点掌握度。

### 13.4 同步原则

- 本地和云端进度按课程隔离。
- 云端同步失败时不能阻断本地学习。
- 冲突时优先保留完成度更高、更新时间更新的数据。
- 进度合并必须避免已完成内容回退为未完成。

## 14. 错题本与复习

### 14.1 收录规则

- 首次回答错误的题目自动进入错题本。
- 用户可以手动收藏任何题目。
- 连续两次复习答对后，可标记为已掌握，但保留历史记录。

### 14.2 复习模式

- 按课程、阶段、题型和知识点筛选。
- 优先出现近期答错和掌握度较低的题目。
- 同一知识点尽量使用不同题型复习，减少死记答案。
- 复习结果更新知识掌握度，但不重复触发大型成就弹层。

## 17. 核心数据模型建议

### 17.1 QuestionSpec

```ts
type QuestionType =
  | "prediction"
  | "implementation"
  | "diagnosis"
  | "repair"
  | "completion"
  | "execution-order"
  | "best-practice"
  | "concept-match"
  | "equivalent-code"
  | "sequence";

interface QuestionSpec {
  id: string;
  type: QuestionType;
  prompt: string;
  materials?: QuestionMaterial[];
  options: QuestionOption[];
  correctOptionId: string;
  explanation: string;
  summaries: string[];
  difficulty: "beginner" | "intermediate" | "advanced";
  estimatedSeconds: number;
}

interface QuestionOption {
  id: string;
  label?: string;
  text?: string;
  code?: string;
  language?: string;
  diffLines?: number[];
  feedback: string;
}
```

### 17.2 LessonSpec 调整

现有 `LessonSpec.questions` 从“至少一道普通选择题”升级为“一个或多个带类型的问题”。课程校验需要增加：

- `type` 必须是受支持题型。
- 每道题至少两个选项。
- 正确选项必须存在。
- 每个选项必须有定向反馈。
- 代码型选项必须声明语言。
- `implementation`、`repair` 等题型必须至少包含一个代码选项。
- 每道题必须具有预计用时和难度。

### 17.3 ProjectSpec

```ts
interface ProjectSpec {
  id: string;
  stageId: string;
  title: string;
  brief: string;
  objectives: string[];
  steps: ProjectStep[];
  finalFiles: LessonFile[];
  summaries: string[];
}

interface ProjectStep {
  id: string;
  title: string;
  context: string;
  question: QuestionSpec;
  execution?: ExecutionSpec;
}
```

## 19. 非功能需求

### 19.1 性能

- 移动网络下优先加载正文、代码和题目，3D 资源延迟加载。
- 首次进入题目时不因 3D 场景阻塞交互。
- 切换课程时及时释放 Canvas、事件监听和动画资源。
- 代码选项展开、切换和提交需要保持即时反馈。

### 19.2 可访问性

- 支持键盘操作和清晰焦点状态。
- 选项状态不能仅通过颜色表达。
- 支持 `prefers-reduced-motion`。
- 代码、正文和反馈满足基本对比度要求。
- 动画结束后，关键信息必须以文字形式保留。

### 19.3 兼容性

- 重点支持现代 Chrome、Safari、Edge 和移动端浏览器。
- WebGL 不可用时功能完整降级。
- 360px 宽度下可以完成全部核心学习流程。

### 19.4 安全与隐私

- 不在客户端暴露 Supabase 服务端密钥。
- 学习数据按用户隔离并使用行级安全策略。
- 不执行用户提交的任意代码。
- 日志中不记录敏感认证信息。

## 21. 验收标准

### 21.1 多题型系统

- 现有课程无需一次性重写即可继续运行。
- 同一知识点可以顺序展示至少两道不同题型的问题。
- `implementation` 题可以展示不少于两个代码选项。
- 每个选项提交后都能显示对应反馈。
- 只有完成规定问题后才写入知识点完成状态。

### 21.2 移动端

- 用户可以展开、收起、选择和重新作答代码选项。
- 无 hover 的触屏设备可以使用全部核心功能。
- 3D 降级不影响答题、反馈、终端和总结。

### 21.3 数据与兼容

- 旧版 localStorage 进度能够安全读取或迁移。
- Node.js 与 Next.js 进度继续隔离。
- 损坏进度不会导致页面崩溃。
- 登录同步失败时，本地学习仍可继续。

### 21.4 工程质量

- 课程目录校验通过。
- 单元测试和组件测试通过。
- TypeScript 类型检查通过。
- ESLint 检查通过。
- Next.js 生产构建通过。
- 关键移动端尺寸完成手动验收。

## 23. 风险与应对

| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| 题型数量增加导致内容制作成本快速上升 | 课程更新速度下降 | 先建立题型模板，每个知识点按需要选择题型，不强制全部覆盖 |
| 代码选项在手机端难以比较 | 用户阅读负担过高 | 折叠相同上下文、突出差异行、支持全屏查看 |
| 用户通过记忆选项而非理解答题 | 学习效果下降 | 同一知识点使用不同题型、随机化选项顺序、加入迁移题 |
| 3D 效果影响性能 | 移动端卡顿或退出 | 延迟加载、尺寸检测、减少动态效果和 2D fallback |
| 游戏化效果喧宾夺主 | 干扰代码阅读 | 强反馈只用于关键完成节点，常态界面保持稳定 |
| 旧课程数据难以一次性迁移 | 开发周期延长 | 保留兼容适配层，先试点再批量迁移 |
| 云端同步冲突导致进度回退 | 用户失去信任 | 采用单调合并原则，完成状态不得回退 |

## 24. 产品原则

- 让用户始终处于“阅读—判断—反馈—理解”的主动循环。
- 手机端不是桌面 IDE 的缩小版，应围绕轻操作重新设计学习方式。
- 选择题的价值来自高质量干扰项和定向反馈，而不是选择动作本身。
- 每道题都应对应一个明确知识目标。
- 阶段项目必须显式复用前面知识，而不是把普通题目简单变长。
- 3D 和粒子效果必须表达学习语义，不能遮挡核心内容。
- 错误是学习数据，不是惩罚用户的理由。
- 在继续扩充课程数量之前，优先验证题型深度和学习完成率。

## 25. 最终产品愿景

NodePath 最终不是一个要求用户在手机上艰难敲代码的网站，而是一个帮助用户真正读懂代码、理解运行机制、判断实现方案并完成项目决策的移动编程学习平台。

用户可以利用碎片时间完成一个知识点，通过多种代码选择题发现自己的理解偏差，再借助 3D 场景、粒子流、终端日志和知识总结看到代码为什么这样运行。随着学习推进，零散知识点逐渐连接成完整技术路径，最终形成可以迁移到真实开发中的编程心智模型。
