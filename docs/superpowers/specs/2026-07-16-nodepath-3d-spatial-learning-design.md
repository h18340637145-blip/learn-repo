# NodePath 3D 空间化学习体验设计

## 背景

NodePath 当前已经具备完整的 Node.js 学习路径、答题触发、运行帧、日志输出、阶段项目和进度保存能力。随着课程数量增长到完整学习体系，现有全局课程切换器会把大量课程平铺在页面中，导航压力变大，学习路径的探索感也会变弱。

本设计目标是在不破坏现有学习闭环的前提下，把 Node.js 学习过程升级为更具空间感、粒子感和沉浸感的体验。学习者仍然通过「讲解 → 代码 → 答题 → 运行可视化 → 总结」掌握知识，但课程入口和运行反馈会更像一个可进入的运行时宇宙。

## 已确认方向

采用「阶段星图 + 重点 3D 实验舱」方案。

- 左侧导航只保留学习阶段入口，不再展示全部课程。
- 进入阶段后，主区域以空间星图展示本阶段知识点和阶段项目。
- 课程内容仍保留清晰的讲解、代码、选项、日志和总结。
- 答对后优先使用 Three.js 展示运行过程，重点课程配置主题化 3D 场景。
- 桌面端优先完整沉浸，移动端和低性能设备自动降级。

## 目标

1. 提高学习吸引力，让学习者感觉是在探索一个可进入的 Node.js 运行时空间。
2. 降低完整课程体系下的导航拥挤感，避免把 80+ 节课程全部铺在导航栏。
3. 用 3D 和粒子效果解释运行机制，而不是只做装饰背景。
4. 保持现有答题、运行帧、日志、总结和进度保存逻辑可用。
5. 建立可扩展的可视化描述模型，后续课程可以按数据配置视觉器。

## 非目标

第一期不实现以下内容：

- 任意用户代码的真实 Node.js 沙箱执行。
- 用户账号、云端进度或 Supabase 持久化。
- 每一节课都定制独立 3D 场景。
- 重型后处理视觉系统。
- 新增课程内容本身。

## 学习路径与导航

### 阶段入口

左侧导航从「阶段 + 展开课程列表」改为「阶段入口」。每个阶段展示以下信息：

- 阶段编号。
- 阶段标题。
- 阶段完成度。
- 当前、已完成、锁定或计划状态。

点击阶段后，主内容区切换到该阶段的空间页。空间页展示本阶段 8 个知识点和 1 个阶段项目，视觉上使用星图、轨道、节点和进度能量线表达学习路径。

### 课程选择

课程不再通过全局 `lesson-switcher` 平铺展示。进入某个阶段后，只展示当前阶段内的课程节点。点击节点进入现有学习模式。

现有 `lessonIndex` 仍负责打开具体课程，但需要新增 `selectedStageId` 或等价状态来表示当前阶段。课程切换范围从全局课程列表收敛到当前阶段内。

### 项目入口

阶段项目作为阶段星图中的特殊节点展示。完成阶段知识点后，阶段项目节点应更突出，但第一期不强制实现硬锁定；可以延续现有可直接进入的行为，并通过状态提示学习顺序。

## 3D 运行可视化

### 两层 3D 架构

第一层是全局学习空间。它负责阶段星图、课程节点、进度能量线和空间背景。它可以升级或包裹现有 `ImmersiveBackdrop`，但不应遮挡阅读和答题区域。

第二层是课程运行实验舱。答对后，`Runtime Visualizer` 根据课程主题展示不同 3D 场景，同时继续显示日志输出、执行步骤和总结。

### 重点 3D 场景

第一期重点覆盖以下主题：

| 主题 | 视觉模型 | 学习价值 |
| --- | --- | --- |
| HTTP 基础 | 请求粒子穿过 Header、Router、Handler、Response 管线 | 理解一次网络事务的流转顺序 |
| API 与服务设计 | 资源节点、验证闸门、错误模型和日志轨道 | 理解服务边界和可靠接口结构 |
| 进程与并发 | 主线程、libuv 线程池、Worker 和 IPC 通道 | 理解并发模型的空间关系 |
| 实时通信 | 连接星网、心跳脉冲、广播扩散和恢复路径 | 理解实时系统的连接生命周期 |
| 测试与安全 | 断言光束、安全边界、密钥隔离和风险层 | 理解质量保障与安全边界 |
| 诊断与生产工程 | CPU 热区、Heap 空间、GC 波纹、火焰图塔和发布检查流 | 理解性能诊断和生产运维流程 |

未配置主题化场景的课程使用通用粒子管线，保证课程扩展不被 3D 资产阻塞。

### 数据驱动配置

课程数据增加 `visualizer` 描述字段，由统一的视觉器注册表解析。页面组件不应为每节课堆叠硬编码条件。

推荐字段示例：

```ts
visualizer: {
  type: "http-pipeline",
  emphasis: "request-response",
  nodes: ["Client", "Server", "Router", "Handler", "Response"]
}
```

`visualizer.type` 决定使用哪个 3D 场景。运行时组件读取当前课程的 `status`、`frame`、`lanes` 和 `visualizer`，驱动粒子、节点高亮、连线和阶段变化。

## 技术边界

### 依赖

第一期建议引入以下依赖：

- `three`
- `@react-three/fiber`
- `@react-three/drei`

暂不引入重型后处理库。只有在实现中确认必要时，再评估额外依赖。

### Next.js 组件边界

项目使用 Next.js 16 App Router。实现需要遵循以下边界：

- `app/layout.tsx` 保持 Server Component，用于静态元数据。
- 3D 场景、答题交互、运行帧播放等浏览器能力放在 Client Component 中。
- Three.js 相关组件应通过客户端边界和动态加载接入，避免影响服务端元数据和首屏结构。
- 现有进度、答题状态和运行状态继续放在 `app/learning-studio.tsx` 或拆出的 Client 子组件中。

### 推荐模块划分

第一期可按以下方向拆分：

- `components/learning-space/`：阶段星图、课程节点、学习空间外壳。
- `components/visualizers/`：运行可视化组件和视觉器注册表。
- `components/visualizers/scenes/`：HTTP、并发、实时通信、测试安全、诊断生产等主题场景。
- `components/visualizers/fallbacks/`：无 WebGL、低性能和减少动态效果时的替代展示。
- `content` 或 `lib/curriculum`：扩展课程 `visualizer` 类型定义。

## 性能与降级

### 性能策略

桌面端默认启用完整空间场景。移动端或低性能设备降低视觉负载：

- 限制设备像素比（DPR）。
- 减少粒子数量。
- 关闭复杂后处理。
- 降低持续动画频率。
- 页面隐藏时暂停动画。

### 降级策略

以下情况使用轻量版本：

- WebGL 不可用。
- 用户开启 `prefers-reduced-motion`。
- 设备性能不足。
- 移动端视口空间不足。

轻量版本仍需表达运行顺序和状态变化，可以使用 CSS、Canvas 或现有 lane 视图，不得只显示空白区域。

## 交互与可访问性

3D 场景负责解释运行关系，DOM 负责核心学习交互。代码、题目、选项、日志和按钮必须保持可读、可点击、可聚焦。

要求如下：

- 3D Canvas 不遮挡答题按钮和代码区域。
- 阶段节点和课程节点需要有文本标签。
- 动画不能造成文本重叠或布局跳动。
- 移动端布局不能依赖悬停交互。
- 减少动态效果模式下仍能完成课程学习。

## 验收标准

1. 左侧导航只展示阶段入口，不再平铺全部课程。
2. 主内容区可进入阶段空间页，并只展示当前阶段课程。
3. 点击课程节点可以进入现有学习流程。
4. 答题、运行帧、日志、总结和进度保存继续可用。
5. 重点主题课程可展示对应 3D 运行场景。
6. 未配置主题场景的课程可以使用通用粒子管线或轻量 fallback。
7. 桌面端 Canvas 非空、动画可见、构图正确。
8. 移动端布局不重叠，按钮和文字不溢出。
9. WebGL 不可用或减少动态效果时仍可完成学习。
10. 通过 `npm run lint`、`npm run build` 和 `git diff --check`。

## 自检结果

- 占位符：无未完成标记、未决问题或空章节。
- 一致性：导航、课程选择、3D 可视化和性能降级均服务于「A + 重点 B」方向。
- 范围：第一期聚焦结构改造、可复用 3D 系统和重点课程示范，可以用一份实现计划覆盖。
- 模糊性：未要求所有课程独立定制 3D 场景，未要求真实执行用户代码，避免范围膨胀。
